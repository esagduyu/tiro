# Tiro

## What This Is

Tiro is a local-first, open-source, model-agnostic reading OS for the AI age. It saves web pages and email newsletters as clean markdown, enriches them with AI-extracted metadata, and uses Claude Opus 4.6 for deep cross-document reasoning — daily digests, trust analysis, and learned reading preferences. Everything runs locally. The user owns their data.

Named after Cicero's freedman who preserved and organized his master's works for posterity. *"...without you the oracle was dumb." — Cicero to Tiro, 53 BC*

**Context:** Born at the "Built with Opus 4.6: Claude Code Hackathon" (Feb 10–16, 2026); the frozen MIT-licensed hackathon build lives at github.com/esagduyu/project-tiro — THIS repo (github.com/esagduyu/tiro, public, branch `main`) is the continuation. PROJECT_TIRO_SPEC.md is the historical hackathon spec; PRODUCT_ROADMAP.md is the forward-looking source of truth.

## Architecture

```
Web UI (FastAPI serves HTML/JS at localhost:8000)
  ↕ REST API
FastAPI Backend (Python)
  ├── Ingestion Engine (readability-lxml + markdownify)
  ├── Intelligence Layer (Opus 4.6 — digests, analysis, preferences)
  ├── Lightweight Processing (Haiku — tags, entities, summaries)
  ├── Query Layer (ChromaDB semantic search + SQLite metadata)
  └── MCP Server (exposes knowledge base to Claude)
  ↕
Storage Layer (all local)
  ├── articles/*.md (markdown files with YAML frontmatter)
  ├── tiro.db (SQLite — metadata, preferences, stats)
  ├── chroma/ (ChromaDB — vector embeddings)
  └── config.yaml (user configuration)
```

**MCP server:** `tiro/mcp/server.py` exposes the library to Claude Desktop and Claude Code via 10 tools (see below). Client config needs `TIRO_API_TOKEN` (when a password is set) and `TIRO_CONFIG` (absolute config path) in its env block.

## Tech Stack

- **Backend:** FastAPI, uvicorn, Python 3.11+
- **Content extraction:** readability-lxml, markdownify
- **Email parsing:** Python email stdlib → readability-lxml → markdownify
- **Storage:** SQLite (metadata), ChromaDB (vectors), markdown files on disk
- **Embeddings:** sentence-transformers (all-MiniLM-L6-v2) locally
- **AI (heavy):** Claude Opus 4.6 via Anthropic API (digests, analysis, preferences)
- **AI (light):** Claude Haiku 4.5 via Anthropic API (tags, entities, summaries)
- **Frontend:** Minimal HTML/CSS/JS served by FastAPI (Jinja2 templates)
- **MCP:** Python MCP SDK

## Key Conventions

- **Use `uv` for all Python version and dependency management** — never use pip directly. Use `uv pip install`, `uv venv`, `uv run`, etc. Dependencies defined in pyproject.toml.
- Python 3.11+, async throughout (async def for all route handlers)
- Use httpx for async HTTP calls
- Use python-frontmatter for reading/writing markdown with YAML frontmatter
- All files stored under a configurable `library_path` (default: `./tiro-library/`)
- Structured Anthropic API responses: always request JSON output, parse with error handling
- Logging via Python logging module, INFO level default
- Graceful error handling everywhere — never crash the server on bad input
- See PROJECT_TIRO_SPEC.md § "Data Models" for the full SQLite schema, markdown format, and ChromaDB collection spec
- See PROJECT_TIRO_SPEC.md § "Key Prompt Templates" for all Opus/Haiku prompt templates
- See PROJECT_TIRO_SPEC.md § "API Endpoints" for the full endpoint list

## Quick Start

```bash
uv sync                    # Install dependencies (creates venv automatically)
uv run tiro init           # Creates config.yaml, library, prompts for API key
uv run tiro run            # Start server on localhost:8000 (auto-opens browser)
                           # First launch: /login asks you to SET a password (required)
uv run tiro run --lan      # Bind to 0.0.0.0 for LAN access — refuses without a password
```

Other CLI verbs: `tiro set-password`, `tiro token create|list|revoke <name/id>`, `tiro doctor [--fix] [--json]` (server stopped), `tiro audit [--date|--month|--service|--json]`, `tiro status`, `tiro delete <id>`, `tiro export`, `tiro import-emails <dir>`, `tiro setup-email`, `tiro check-email`, `tiro migrate`, `tiro backup [--output|--include-audio]`, `tiro restore <snapshot> [--yes]`, `tiro import <bundle> [--conflicts]`. Tests: `uv run pytest` (432, must stay 0-warnings).

Before starting the server, kill any existing process on port 8000:
```bash
lsof -ti :8000 | xargs kill -9
```

Content reaching `process_article()` is already sanitized. Client renders markdown through `renderMarkdown` (marked → DOMPurify) and escapes every server string at `innerHTML` sinks via `esc()`/`num()`. Never route the already-clean settings token UI through a sanitize+innerHTML path.

Frontend auto-generates on first digest visit by POSTing when the GET 404s. `GET /api/articles/{id}/audio` stays GET by design (`audio.src` issues GETs). Known cosmetic gap: cached digest markdown keeps dead `/articles/N` links after an article is deleted (accepted, M4b).

Residual inconsistencies (file-without-row, row-without-file) are recoverable and are `tiro doctor`'s job (M5).

Run with the server stopped; `--fix --json` prints post-fix state; exit 0 clean / 1 issues-remain. ALL config.yaml writes — server routes AND CLI (init API-key saves, setup-email) — go through `persist_config(config, updates)` (tiro/config.py: ruamel round-trip preserving comments, atomic temp+replace, chmod 0600, path from `config.config_path`) — never write config.yaml directly (grep-gated: zero `Path("config.yaml")` and zero `write_text(yaml.dump` in tiro/, both test-enforced). Stats increments are transition-gated (`is_read` 0→1, rating NULL→value; `opened_count` counts every open); `update_stat` has a field allowlist. IMAP sync task lives on `app.state.imap_task` with dynamic restart from `POST /api/settings/email`; disabling both email features is valid (turns email off). - **Sources/authors/views (M1.2):** Source deletion (`DELETE /api/sources/{id}`) loops through the `delete_article` lifecycle coordinator per article, with an `auto_backup(config, "source-delete")` snapshot taken first — never a raw cascade `DELETE`. `POST /api/sources/merge` repoints articles' `source_id`, ORs the VIP flag into the surviving source, and 409s (`type_mismatch`) when `source_type` differs unless `force`. Authors are a *derived* VIP/merge layer over `articles.author` (free text, still the authoritative value for display/frontmatter, never rewritten) — the `authors`/`article_authors` tables (migration 007) dedupe on **exact-canonical `canonical_key` only** (fuzzy/near-duplicate matching is out of scope, deferred per Decisions), backfilled by the migration and kept in sync at ingest via `tiro.authors.ensure_author`/`link_article_author`. Author VIP ORs with source VIP as an input to both decay weights (`tiro/decay.py`, VIP source OR VIP author ⇒ slowest decay) and digest ranking context (`tiro/intelligence/digest.py` gathers `vip_authors` alongside `vip_sources`) — not a separate immunity path. Saved views (`saved_views` table) persist the filter-panel state as the raw `filter_json` string the client sent (server only validates it `json.loads()`s to a dict — no server-side schema, so filter shapes stay forward-compatible) plus `sort_mode` and an integer `position`; capped at `MAX_SAVED_VIEWS = 20` (400 past the cap); reordering is two client-driven PATCH calls swapping `position` — **position buttons, not drag-and-drop** (deferred per Decisions). `/sources` page (source + author management UI) follows the existing settings-page patterns (status-style cards, modals, toast feedback). `tiro doctor --fix` vacuums unreferenced `authors` rows the same way it already vacuums tags/entities. **Env config overlay & Docker**: `tiro/config.py` `_apply_env_overlay()` lets `TIRO_<FIELD_UPPER>` env vars override every `TiroConfig` field except `config_path` (precedence env > yaml > defaults; bool/int/float coercion from the field's declared type) — this is the Docker configuration path (`Dockerfile`/`deploy/docker/docker-compose.yml` set `TIRO_LIBRARY_PATH`/`TIRO_HOST`; see README "Docker" section for the rest). There is no `password` field to overlay — `TIRO_PASSWORD` is unsupported/does nothing; setting the password stays CLI-only via `tiro set-password`. `run.py`'s `_config_path()` now mirrors `tiro/mcp/server.py`'s and honors `TIRO_CONFIG` (absolute path) instead of hardcoding `./config.yaml` — a real trap that bit a prior session (regression-tested: `test_run_py_config_path_honors_tiro_config`). Docker publishes no image (build-it-yourself `Dockerfile` only, no registry push in CI).
- **Library Wiki (P1b/W1):** `{library}/wiki/**/*.md` (one page per entity/tag node, frontmatter contract: uid/kind/title/entity_type/status/article_uids/source_count/generated_by/updated_at/user_pinned_note) is files-as-truth — `tiro/wiki.py`'s derived `wiki_pages`/`wiki_page_articles` tables (migration 008) are a queryable cache, reconciled files-win by `reconcile_wiki_index()`, which is `tiro doctor`'s job (wired into its drift report + `--fix`), never authoritative for content. Generation (`tiro/wiki_gen.py`, on-demand only in W1 via `POST /api/wiki/generate` and `POST /api/wiki/{slug}/regenerate` — W2 adds a scheduled background sync) is built from exactly three ingredients: the gathered articles' summaries/trust-signals, the (user-editable) `_schema.md` instructions, and the page's OWN prior body — it never reads another wiki page's content (no wiki-reads-wiki, by design, cross-page context is out of scope for W1). Citations are mandatory, not cosmetic: `[[stem|label]]` targets in the model's output are resolved against the gathered articles' `markdown_path` stems; if ZERO resolve, generation is discarded outright (no file, no derived rows, no log.md line — the one hard trust invariant is a wiki page can never cite nothing real); a page with some unresolved alongside resolved citations is fine, `write_page`'s `article_uids` only gets the resolved subset. `user_pinned_note` always survives regeneration (read from the prior page, threaded through `write_page` even on a from-scratch regenerate). Staleness is free SQL, not an LLM call: `mark_pages_stale()` runs inside the ingest transaction and matches purely by SLUG (`entities/{wiki_slugify(name)}` / `concepts/{wiki_slugify(name)}`), not title, so it's immune to cosmetic title changes. `_schema.md` is copied from a packaged default on first use (`ensure_schema_file()`) and never overwritten again — user-owned from that point on. Wikilinks (`[[stem|label]]`) resolve client-side via the citation map `GET /api/wiki/{slug}` returns (`{stem: article_id}`, built server-side against `articles.markdown_path`); an unresolvable stem renders as plain text, never a dead/error link. `wiki/` is included in both `GET /api/export` and `tiro backup` via `wiki_dir.rglob("*.md")` (mirrors `articles/`). Page routes: `/wiki` (list, grouped by kind) and `/wiki/{slug}` (page view + Regenerate button), both `require_page_auth`-gated same as `/sources`/`/graph`.
